📊 RetailCRM Analytics Dashboard
Система автоматизации сбора данных, мониторинга сделок и визуализации аналитики в реальном времени.
🎯 Цель проекта
Построить отказоустойчивый ETL-пайплайн для синхронизации заказов из RetailCRM в Supabase (PostgreSQL), настроить мгновенные уведомления о VIP-сделках и создать интерактивный дашборд для анализа ключевых метрик.
🛠 Технологический стек
Backend: Node.js (ETL-скрипты), RetailCRM API v5, Telegram Bot API.
Database: PostgreSQL (Supabase) + Row Level Security (RLS).
Frontend: Next.js (React), Recharts (визуализация данных).
DevOps: GitHub, Vercel, ESLint + Prettier.
🚀 Ключевые этапы реализации
Проектирование и аудит API: Проведен технический аудит API RetailCRM, адаптирована архитектура под реальные справочники статусов, что исключило ошибки 400/404.
ETL-пайплайн: Реализована синхронизация данных с использованием метода .upsert(), обеспечивающего идемпотентность (защиту от дублей) и целостность базы данных.
Бизнес-логика (VIP-мониторинг): Разработан триггер на Node.js, отправляющий уведомления в Telegram при обнаружении заказов на сумму свыше 50 000 ₸.
Визуализация: Построен дашборд на Next.js с динамическими графиками выручки и количества заказов.
🛠 Troubleshooting & Engineering Challenges (Deep Dive)
В ходе работы над проектом были успешно решены критические технические проблемы, связанные с кроссплатформенной разработкой и CI/CD процессами:
1. Безопасность и Git (Encoding Issue)
Проблема: Файл .gitignore игнорировался системой Git, создавая риск утечки ключей .env.
Причина: При создании в среде PowerShell файлу была присвоена кодировка UTF-16 LE, которую Git не мог распарсить.
Решение: Выполнена принудительная перекодировка в UTF-8 без BOM. Это обеспечило корректную работу игнорирования системных файлов и секретов.
2. Стабилизация CI/CD окружения (ESLint Conflict)
Проблема: Ошибка Circular structure to JSON при сборке проекта в облаке.
Причина: Конфликт версий между ESLint 9 (flat config) и стандартными плагинами Next.js.
Решение: Проведен даунгрейд и фиксация версий до стабильного стека (ESLint 8.57 + Next 14.2). Это позволило внедрить бесперебойную проверку качества кода перед деплоем.
3. Кроссплатформенная совместимость (Windows vs Linux)
Проблема: Ошибки парсинга файлов конфигурации при деплое на Vercel (Linux) из-за различий в окончаниях строк и кодировках.
Решение: Весь проект переведен в формат UTF-8, а в Prettier настроен принудительный формат LF (endOfLine: "lf"). Это гарантировало 100% стабильность сборки в любой среде.
4. Оптимизация сборки (Build-time Constraints)
Проблема: Ошибки пререндеринга страниц из-за отсутствия доступа к Runtime-переменным окружения на этапе сборки.
Решение: Реализованы проверки инициализации Supabase-клиента и временный перевод критических узлов на клиентский рендеринг (CSR), что обеспечило успешное прохождение пайплайна сборки.
📦 Как запустить проект
Клонирование репозитория:
code
Bash
git clone https://github.com/ВАШ_ЛОГИН/retail-crm-dashboard.git
cd retail-crm-dashboard
Установка зависимостей:
code
Bash
npm install
Настройка окружения:
Создайте файл .env на основе .env.example и заполните ключи API.
Запуск:
Линтинг и форматирование: npm run check
Запуск дашборда: npm run dev
Запуск ETL-синхронизации: node sync.js
