# 📊 RetailCRM Analytics Dashboard

Система автоматизации сбора данных, мониторинга сделок и визуализации аналитики в реальном времени.

---

### 🎯 Цель проекта
Построить отказоустойчивый **ETL-пайплайн** для синхронизации заказов из **RetailCRM** в **Supabase (PostgreSQL)**, настроить мгновенные уведомления о 50+T-сделках и создать интерактивный дашборд.

---

### 🛠 Технологический стек

*   **Backend:** Node.js, RetailCRM API v5, Telegram Bot API.
*   **Database:** PostgreSQL (Supabase) + Row Level Security (RLS).
*   **Frontend:** Next.js (React), Recharts.
*   **DevOps:** GitHub, Vercel, ESLint + Prettier.

---

### 🚀 Ключевые этапы реализации

*   **Проектирование и аудит API:** Проведен технический аудит API RetailCRM, адаптирована архитектура под реальные справочники статусов.
*   **ETL-пайплайн:** Реализована синхронизация данных с использованием метода `.upsert()`, обеспечивающего защиту от дублей.
*   **Бизнес-логика:** Разработан триггер на Node.js для уведомлений в Telegram при заказах свыше **50 000 ₸**.
*   **Визуализация:** Построен дашборд на Next.js с динамическими графиками выручки.

---

### 🛠 Troubleshooting & Engineering Challenges (Deep Dive)

В ходе работы были решены критические проблемы, связанные с кроссплатформенной разработкой:

#### 1. Безопасность и Git (Encoding Issue)
*   **Проблема:** Файл `.gitignore` игнорировался системой, секреты `.env` попадали в индекс.
*   **Причина:** Кодировка **UTF-16 LE** (PowerShell). Git распознает правила только в **UTF-8**.
*   **Решение:** Принудительная перекодировка в UTF-8 без BOM. Это восстановило безопасность проекта.

#### 2. Стабилизация CI/CD (ESLint Conflict)
*   **Проблема:** Ошибка `Circular structure to JSON` при сборке.
*   **Причина:** Конфликт ESLint 9 с конфигами Next.js.
*   **Решение:** Даунгрейд и фиксация до **ESLint 8.57 + Next 14.2**. Это обеспечило стабильный линтинг.

#### 3. Кроссплатформенность (Windows vs Linux)
*   **Проблема:** Ошибки парсинга на Vercel (Linux) файлов, созданных на Windows.
*   **Решение:** Настроен Prettier с принудительным форматом **LF** (`endOfLine: "lf"`) для всех сред.

#### 4. Оптимизация сборки
*   **Проблема:** Билд падал из-за отсутствия доступа к API-ключам на этапе пререндеринга.
*   **Решение:** Внедрены проверки инициализации и временный переход на **CSR** (Client Side Rendering) для стабильности пайплайна.
